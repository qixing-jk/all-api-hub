# Safari Extension Installation Guide

This guide explains how to install the All API Hub extension in Safari browser.

## Quick Difference

- Without a paid Apple Developer Program account: you can still build and enable the extension on your own Mac with Xcode for development or personal use. It is generally not suitable for distribution, and local non-distributed builds may require `Allow Unsigned Extensions` in Safari's Develop menu.
- With a paid Apple Developer Program account: you can properly sign the app/extension and distribute it through TestFlight or the App Store, which is the right path for other users to install it.

## System Requirements

- macOS 11.0 Big Sur or later
- Safari 14.0 or later
- Xcode 13.0 or later (for building)

## Installation Methods

Safari is currently best used through one of these two paths:

1. Build from source, then run it with Xcode
2. Download the prebuilt Safari Xcode bundle from GitHub Releases and open it directly in Xcode

> **Recommendation**
> If you just want to get it running on your own Mac as quickly as possible, start with "Method 2: Use the Release bundle".
> If you want to modify code, debug, or verify local changes, use "Method 1: Build from source".

### Method 1: Build from Source

#### 1. Get the source and build Safari output

```bash
# Clone or download the project source
git clone https://github.com/qixing-jk/all-api-hub.git
cd all-api-hub

# Install dependencies
pnpm install

# Build Safari version
pnpm run build:safari
```

After building, the compiled Safari extension will be available in `.output/safari-mv2/`.

#### 2. Generate an Xcode project with the Safari converter

```bash
xcrun safari-web-extension-converter .output/safari-mv2/
```

If you want to control the output location, app name, or bundle identifier, you can also run:

```bash
xcrun safari-web-extension-converter .output/safari-mv2/ \
  --project-location /path/to/all-api-hub-safari-project \
  --app-name "All API Hub" \
  --bundle-identifier "io.github.qixingjk.allapihub"
```

This generates the Xcode project that hosts the Safari extension.

#### 3. Build and run in Xcode

1. Open the generated Xcode project
2. Make sure your Mac is selected as the target device
3. Click `Product > Run`, or press `Cmd + R`
4. On first run, Xcode will ask you to resolve signing; without a paid account you can usually use your `Personal Team` for local testing
5. After a successful build, Safari will prompt you to enable the extension

#### 4. Enable the extension in Safari

1. Open Safari
2. Click `Safari > Settings`
3. If this is a local non-distributed build, enable `Allow Unsigned Extensions` in the `Develop` menu
4. Open the `Extensions` tab
5. Find `All API Hub` and enable it
6. Grant permissions as needed

### Method 2: Use the Safari Xcode bundle from GitHub Releases

#### 1. Download the correct Safari asset

Open the Releases page and download a file with a name like:

```text
all-api-hub-<version>-safari-xcode-bundle.zip
```

For example:

```text
all-api-hub-3.29.0-safari-xcode-bundle.zip
```

Do not download only `all-api-hub-<version>-safari.zip`.

WHY:
`all-api-hub-<version>-safari.zip` is only the compiled Safari extension output. It does not include the Xcode project you need for a direct open-and-run flow. The asset meant for that workflow is `safari-xcode-bundle.zip`.

#### 2. What is inside the extracted bundle

After extracting `all-api-hub-<version>-safari-xcode-bundle.zip`, you will typically see all of the following together:

- `all-api-hub-<version>-safari.zip`
- `safari-mv2/`
- the Xcode project directory generated by the converter

This keeps the compiled Safari files and the Xcode project together so the project can resolve its referenced extension files after extraction.

#### 3. Open the bundled Xcode project directly

1. Locate the Xcode project inside the extracted bundle
2. Open it in Xcode
3. Make sure your Mac is selected as the target device
4. Click `Product > Run`
5. When Safari prompts you, enable the extension in `Safari > Settings > Extensions`

#### 4. When the other bundled files matter

- `safari-mv2/`: the compiled Safari extension directory referenced by the Xcode project
- `all-api-hub-<version>-safari.zip`: useful if you want to archive, inspect, or redistribute the compiled Safari build itself

In most cases, you should keep the extracted bundle intact instead of moving only one subdirectory elsewhere.

### Method 3: Temporary Debug Loading (Development Only)

Some macOS / Safari versions allow temporary debug loading, but it is not a proper installation or distribution path:

```bash
pnpm run build:safari
```

Then enable Safari developer mode:

1. Open `Safari > Settings > Advanced`
2. Check "Show Develop menu in menu bar"
3. Click `Develop > Allow Unsigned Extensions`
4. Enable the extension in `Safari > Settings > Extensions`

> **Note**
> If this path is unavailable, use the Xcode flow above. Proper distribution should still go through signed Apple distribution channels.

## Development Debugging

### Development Build

```bash
# Development build with hot reload
pnpm run dev -- -b safari
```

### Debugging Extension

1. **Debug background script/popup**:
   - In Safari, right-click the extension icon
   - Select `Inspect` or open Web Inspector

2. **Debug content scripts**:
   - On any webpage, right-click the page
   - Select `Inspect Element`
   - Check extension-related logs in the console

## FAQ

### Q: Why does Safari require special handling?

A: Safari extensions must be packaged as macOS apps for installation and distribution, unlike Chrome/Edge/Firefox which can directly install `.crx` or `.xpi` files.

### Q: What's the difference between having and not having a developer account?

A:
- No paid account: good for local development or personal use on your own Mac, but not proper end-user distribution.
- Paid account: required for proper signing and distribution through TestFlight / App Store.

### Q: Can I install it like Chrome?

A: No. Safari doesn't support normal production-style unpacked installs like Chrome. Local use typically goes through Xcode, while real distribution goes through TestFlight / App Store.

### Q: What if I encounter build errors?

A: Make sure:
- Xcode command line tools are installed: `xcode-select --install`
- Xcode license is accepted: `sudo xcodebuild -license accept`
- Node.js version >= 18

### Q: Are there differences from the Chrome version?

A: Basic functionality is identical. However, due to Safari WebExtensions API limitations, some features may differ:
- `sidePanel` API is not available in Safari (uses popup window instead)
- Some permission request methods may differ

### Q: How do I update the extension?

A:
- If you installed from source: rebuild the Safari output and rerun the Xcode project
- If you installed from Releases: download the new `safari-xcode-bundle.zip`, extract it, and run the new bundled Xcode project

## Uninstallation

1. Open Safari
2. Go to `Safari > Settings > Extensions`
3. Uncheck `All API Hub`
4. Delete the Xcode-generated macOS app
